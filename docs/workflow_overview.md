# Workflow Overview

## Purpose

This document gives a public-safe overview of the `Research` workflow as a
workflow pack.

It is meant to answer four simple questions:

1. what problem this workflow is trying to prevent
2. what the workflow broadly does
3. what the current sealed route looks like
4. what this repository does and does not claim about that route

## Problem Surface

This workflow is organized around a workflow-governance problem rather than
only an output-quality problem.

The main problem family is:

- `silent workflow-governance drift`

Related failure names carried by the current stack are:

- `irreconstructible execution`
- `unbounded movement`

The target properties are therefore:

- drift externalization
- reconstructability
- bounded execution

## What The Workflow Does

At a public-safe level, the `Research` workflow turns a research problem into a
bounded, attributable, and reviewable research route.

Its broad movement is:

1. ground the problem and formalize the question
2. determine knowledge readiness
3. expand and lock a lawful route
4. resolve dependencies and build the route-bound package
5. freeze the execution envelope
6. run pilot and disposition review
7. continue to full analysis only under explicit lawful conditions
8. calibrate claims and assemble submission surfaces
9. pass through internal review, human review, and release-facing QA
10. handle stop, redesign, successor-spawn, reviewer-return, and archive paths
    through typed lawful dispositions

## Four-Layer Reading

The public workflow should be read through four structural layers:

### Law

`Law` states what is allowed, required, blocked, returned, or promoted.

At current scope:

- the workflow uses an authored legal surface rather than runtime guesswork
- the live machine-carried law surface is the promoted current surface
- the repository does **not** claim one separately deployed executable unit per
  `LAW-*` coordinate

### Object

`Object` states what governed thing exists, what identity it carries, and what
surface downstream work is allowed to inherit.

At current public-safe summary level:

- the `Research` workflow currently uses `19` object surfaces
- those surfaces carry identity for intent, question, readiness, route lock,
  build, envelope, pilot, results, claims, submission, review, release,
  reviewer response, successor, and terminal handling

### Runtime

`Runtime` states how movement is enforced.

At current public-safe summary level:

- runtime is expressed through controller surfaces and transition corridors
- controllers govern actions such as intake, route lock, build freeze, pilot
  review, submission, review release, and redesign terminal handling
- corridors govern lawful crossing between major workflow stages

### Audit

`Audit` states how execution remains reconstructable.

At current public-safe summary level:

- audit keeps one explicit coordinate set across the formal units
- replay, compatibility, lineage, and evidence-envelope responsibilities remain
  explicit rather than implied
